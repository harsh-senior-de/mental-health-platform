## Appointment Booking and Payment

### REQ-appointment-booking

source: specs/001-patient-psychiatrist-match/spec.md (FR-011, FR-011a, FR-011b,
FR-011c, FR-011d, FR-011e)
description: Patient selects a slot; a real-time availability check is performed at slot
selection before proceeding to checkout. If the slot is no longer available, an inline
error is shown and the patient stays on the same screen. If available, the slot is held
for up to 10 minutes during checkout. Payment via Razorpay with a three-path confirmation
strategy: Path 1 (primary) — real-time HMAC-SHA256 signature verification; Path 2 —
Razorpay webhook (async backup); Path 3 — scheduled reconciliation job every 15 minutes
querying non-terminal orders older than 10 minutes. If payment is confirmed but booking
cannot be completed, an immediate automatic full refund is issued and the patient is
notified. If no payment confirmation arrives within the 10-minute hold window, the slot
is released (no charge exists). The confirm button is disabled on first click; a
server-side idempotency key scoped to each Razorpay order prevents duplicate charges and
bookings. Upon booking confirmation, a Zoom meeting is created via the Zoom API using the
platform's single Zoom Business account. If Zoom creation fails, the platform retries up
to 3 times; if all retries fail: booking is cancelled, full refund issued, patient and
psychiatrist notified, failure logged as a trackable operational metric. No manual link
fallback.
acceptance criteria:
- Real-time slot availability check is performed at slot selection.
- Slot is held for up to 10 minutes; released if checkout is not completed.
- Only one booking and one charge are ever created per checkout session.
- If payment is confirmed but booking cannot be completed, a full refund is issued
  immediately.
- Zoom meeting is created automatically on booking confirmation with correct duration.
- Zoom creation failure after 3 retries triggers full refund, patient and psychiatrist
  notification, and logs the failure as an operational metric.

## Data Governance

### REQ-data-lifecycle

source: specs/001-patient-psychiatrist-match/spec.md (FR-028, FR-029, FR-030, FR-031,
FR-032)
description: All data deletion is handled by a single centralised Data Lifecycle Service
using a typed job queue. Three deletion trigger types:
- Type 1 On-Demand: patient-initiated from profile settings; 72-hour SLA; patient
  notified via WhatsApp/SMS on completion.
- Type 2 Abandoned Account Cleanup: triggered after 30 days of no login with incomplete
  intake status (single nudge at 48-hour inactivity mark first); 24-hour SLA.
- Type 3 Data Expiry: triggered automatically by daily scan when last platform activity
  is older than 7 years; 24-hour SLA.
All jobs execute the same two-phase pipeline: Phase 1 — PII erasure (name, mobile number,
WhatsApp number, email, payment details, device tokens, raw identifiers in logs/analytics);
Phase 2 — Clinical Record Anonymisation (patient identifiers in intake, transcripts,
recommendations, audit log entries replaced with a pseudonymous ID; anonymised records
retained 7 years from session date). Every job produces an audit entry with no PII.
Audit entry retained permanently. Deletion job queue status (pending, processing,
completed, SLA-breached by type) is exposed to Platform Admins; no PII visible.
Self-service deletion request option is in patient profile settings with clear consequences
labelled; immediate on-screen acknowledgement; WhatsApp/SMS confirmation on completion.
acceptance criteria:
- On-demand deletion is processed within 72 hours of request.
- Abandoned account deletion is processed within 24 hours of 30-day no-login trigger.
- Data expiry deletion is processed within 24 hours of daily scan trigger.
- PII erasure (Phase 1) is permanent and irreversible.
- Clinical records are anonymised (Phase 2), not deleted, and retained for 7 years.
- Raw Zoom transcripts are anonymised, not deleted, when a deletion job runs (FR-015d).
- No PII appears in deletion job audit entries.
- Deletion job queue status is visible to Platform Admins without exposing PII.
